\section{Motivation}
As modern applications increasingly integrate Large Language Models (LLMs), routing user queries to the appropriate model has become a critical challenge. Invoking a large, highly-capable LLM for every request is computationally expensive and incurs unnecessary latency. Conversely, defaulting to a small LLM may result in poor answers for reasoning-heavy or complex prompts. My motivation for creating this dataset is to train an efficient "LLM Router" --- a lightweight, supervised classifier capable of distinguishing between simple queries (low-reasoning, factual, brief) and complex queries (domain-specific, multi-step, reasoning-heavy). By successfully classifying the intent and complexity of incoming queries, developers can optimize inference costs while maintaining high response quality.

\section{Research Questions}
This project explores the feasibility and limitations of using classical machine learning methods versus small neural approaches for intent routing. Specifically, my research questions are:
\begin{enumerate}
    \item \textbf{Model Complexity vs. Performance:} How do simpler, feature-engineered approaches (like TF-IDF with Logistic Regression) compare to transformer-based embeddings (like BERT) when distinguishing query complexity? Are deep learning embeddings necessary for linguistic separability in this context?
    \item \textbf{Impact of Dataset Size \& Density:} How does the amount of training data affect the F1 score of the router? Furthermore, if we apply dimensionality reduction natively across these scales via Principal Component Analysis (PCA), do semantic vectors hold their distinct boundaries better than sparse matrices?
    \item \textbf{Class Imbalance \& Resampling:} In real-world scenarios, complex queries often outnumber generic greeting queries natively. With our human-annotated dataset exhibiting roughly a 3:1 skew, how drastically does performance drop for classical algorithms, and can techniques like SMOTE or class weighting effectively track and map this?
\end{enumerate}

\section{Dataset Documentation}

\subsection{Data Type and Composition}
The dataset consists of \textbf{800 text-based queries} paired with strictly evaluated binary classification labels. 
\begin{itemize}
    \item \textbf{Features:} The raw text of the user query (String).
    \item \textbf{Labels:} 
    \begin{itemize}
        \item \texttt{small-LLM} (0): Simple, concise, single-entity factual queries. (202 instances, $\sim$25\% of the dataset).
        \item \texttt{large-LLM} (1): Complex, multi-step, analytical, or multi-hop reasoning requests. (598 instances, $\sim$75\% of the dataset).
    \end{itemize}
\end{itemize}

\subsection{External Sources and Manual Collection Process}
This dataset mimics real-world interaction logs and is composed of queries adapted from open-source domains. General knowledge and reasoning questions were pseudo-randomly sampled from real-world question-answering databases \textbf{SQuAD} and \textbf{HotpotQA}.

\textit{Data Collection \& Generation Methodology:}
To assemble this dataset while complying with the homework constraint of creating a novel classification challenge, I strictly utilized programmatic web-scraping methodologies. First, a Python script randomly isolated queries directly from the SQuAD (Stanford Question Answering Dataset) and HotpotQA sets into an unlabeled CSV pool locally. To enforce strict objective labeling, I utilized zero-shot prompting via the OpenAI GPT-4o API, systematically evaluating all 800 instances by delineating whether the question logically required relating independent entities or finding an intermediate fact before arriving at the final conclusion (`large`), or if it was a basic zero-reasoning factoid (`small`).

\subsection{Dataset Characteristics \& Constraints}
\begin{itemize}
    \item The dataset features a naturally imbalanced \textbf{$\sim$75\% 'large' / $\sim$25\% 'small' distribution}, providing a highly realistic skew mirroring specialized application ecosystems for our baseline models to evaluate.
    \item Because these are genuine queries pulled from open bases, the dataset authentically preserves grammatical inconsistencies, shorthand, and natural language noise, forcing models to look beyond structural perfection.
\end{itemize}

\section{Proposed Methods}

\subsection{Feature Extraction Models}
\begin{itemize}
    \item \textbf{TF-IDF Vectorizer:} Serves as the classical machine learning baseline. We extract unigram and bigram features, limited to the top 1000 features (\texttt{max\_features=1000}), ignoring common English stop-words.
    \item \textbf{BERT Embeddings:} A deep-learning based feature extractor utilizing the HuggingFace \texttt{all-MiniLM-L6-v2} SentenceTransformer architecture. This maps each text query to a dense, 384-dimensional vector that encapsulates semantic meaning.
\end{itemize}

\subsection{Classification Algorithm}
\begin{itemize}
    \item \textbf{Logistic Regression:} Used as the generalized classification head operating directly upon both feature mappings. We explicitly utilize \texttt{class\_weight='balanced'} to prevent the linear gradient convergence equations from simply guessing the massive 76\% majority class iteratively.
\end{itemize}

\subsection{Evaluation Metrics and Process}
Models were evaluated utilizing rigorous \textbf{5-Fold Cross-Validation} inside Scikit-Learn pipelines to strictly contain data leakage and prevent localized overfitting on the 800 sample dimensions. Primary performance evaluation metrics generated are \textbf{Accuracy}, \textbf{Precision}, \textbf{Recall}, and \textbf{F1-Score}. We heavily utilize \textbf{imblearn SMOTE} (Synthetic Minority Over-sampling Technique) bounded purely over the respective training folds to evaluate synthetic metric recoveries.

\subsection{Error Analysis: Misclassifications}
To better understand the limitations of BERT semantic embeddings against GPT-4's logical annotations, we extracted out-of-fold predictions to identify systemic patterns in false classifications. 

\textbf{False Positives (True Label: Small, Predicted: Large)}
These instances were logically simple but tricked the BERT embeddings into predicting a "complex" routing requirement due to dense vocabulary.
\textit{Examples:}
\begin{itemize}
    \item \textit{"Based in Paris, what philosophical school was formed by a group of Marxist thinkers?"}
\end{itemize}
\textit{Analysis:} The sentence embeddings grouped phrases referring to "Marxist thinkers" and "philosophical schools" intimately with complex reasoning domains, despite the actual query simply asking for historical facts (a single-entity lookup acceptable for a small-LLM).

\textbf{False Negatives (True Label: Large, Predicted: Small)}
These instances successfully tricked the model into evaluating them as simple facts because of their short grammatical constraints, despite requiring multi-hop analysis natively.
\textit{Examples:}
\begin{itemize}
    \item \textit{"Are Genista and Callicarpa in the same family?"}
\end{itemize}
\textit{Analysis:} These questions feature straightforward "Are they" prefixes, causing the classifier to score them similarly to basic trivia. However, answering them natively requires an LLM to perform boolean comparative reasoning (fetching both taxa trees and evaluating the intersection). The simplistic dimension of sentence embeddings fails to map this implicit reasoning overhead.

\section{Discussion}
\textbf{Performance Context \& Insights:}
The baseline results provided excellent counter-intuitive insights. Initially, we anticipated that the deep-learning model (BERT) would heavily outperform the classical approach. However, TF-IDF continually outcompeted BERT overall (0.78 v.s 0.73 F1-Scores respectively). Because we strictly annotated multi-hop reasoning questions as 'large', these logically specific queries inevitably spanned longer lengths and featured distinctly numerous multi-entity terminology strings (such as isolated names and geographical targets from HotpotQA splits). TF-IDF's fundamental unigram recognition efficiently latches onto explicit logical length parameters and keyword rarity contexts. Conversely, the dense 384-dimensional BERT vector inherently compresses subtle multi-hop entity relations out of frame during inference mappings, dropping structural clarity against simplistic single-hop facts during generalized clustering configurations. 

\textbf{Dataset Imbalance Impact:}
The heavily skewed $\sim$75\%/25\% natural sample distribution proved functionally foundational for architectural validation. Before strictly enforcing \texttt{class\_weight='balanced'}, our initial control executions saw both baseline pipelines instantly collapse mathematically; gradients merely voted the majority class identically across the board maximizing generic accuracy thresholds (yielding an immediate, artificial 75\%) while entirely destroying prediction recall for the generic query class. As proven explicitly by our detailed Experiment 3 SMOTE curves, tracking data subset availability mathematically confirmed that enforcing internal metric weighting natively allowed Logistic Regression equations to artificially perceive sparse 0s equivalently inside the loss function, securing the F1 stabilization efficiently without artificially extrapolating vectors.

Furthermore, as seen intricately mapped within Experiment 2 and 4's dimensionality intersection curves, strictly collapsing NLP matrices via PCA reliably and drastically distorts latent intent boundaries. Aggressively restricting both TF-IDF tracking mechanisms and BERT embeddings functionally collapsed baseline scores beneath generalized logic levels, actively confirming high-dimensional spaces are absolutely strictly required to isolate minor language patterns successfully within intent routing logic contexts. 

\textbf{Future Work:}
If provided further resources to expand this project, I intend to experiment with integrating explicit Metadata token mappings (such as \textit{Sentence Word Count} ratios or \textit{Punctuation Entity tracking}) to completely bypass semantic boundaries inside the router whenever explicit sentence lengths instantly imply heavy logical density parameters. Expanding human evaluations out to multi-class mappings (instead of purely binary distributions) involving 'Coding Syntax' flags or 'Mathematics' indicators would similarly increase routing granularity efficiently for highly scaled enterprise clusters.
